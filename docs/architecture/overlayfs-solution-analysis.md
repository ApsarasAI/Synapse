# Synapse 文件系统方案分析

## 问题

[`product.md`](../product/product.md) 中采用文件系统方案时，选择了 `OverlayFS`。这里需要回答两个问题：

1. 为什么当前方案使用 `OverlayFS`
2. 有没有更好的替代方案

---

## 当前采用 OverlayFS 的理由

结合 [product.md](../product/product.md) 和 [tech-design.md](./tech-design.md)，`OverlayFS` 的选择主要基于以下几点。

### 1. 避免重复复制运行时

项目目标是构建高并发、低延迟的代码执行沙箱。Python 运行时和常用库体积较大，如果每次创建沙箱都复制一份完整运行时，会带来明显的启动开销、磁盘 IO 开销和内存压力。

`OverlayFS` 可以将运行时作为共享只读层挂载，每个沙箱只需要准备自己的可写层，而不需要复制整个基础环境。

### 2. 共享只读层的 page cache

多个沙箱共享同一份只读层时，内核可以复用 page cache。这一点对高并发场景非常关键：

- 同一份 Python 解释器和标准库不会被重复加载
- 常用科学计算库也可以共享缓存页
- 在 200 并发目标下，整体内存占用更可控

这也是 `product.md` 中将总内存目标控制在较低水平的重要前提。

### 3. 保留每个沙箱独立可写层

`OverlayFS` 天然支持：

- 共享只读 lowerdir
- 每个沙箱独立 upperdir/workdir
- 执行结束后直接销毁可写层

这非常适合 Synapse 的执行模型：运行时环境尽可能共享，用户代码和临时文件严格隔离，执行后快速回收。

### 4. 启动路径简单，适合 MVP

当前方案的核心准备过程是：

1. 创建空目录
2. 挂载 overlay
3. `pivot_root`
4. 挂载必要的虚拟文件系统

相较于完整容器镜像系统或更重的虚拟化方案，这条路径更短，工程实现复杂度更低，更适合 MVP 阶段快速落地。

### 5. Linux 内核原生支持，成熟度高

`OverlayFS` 是 Linux 内核原生文件系统能力，生态成熟，使用方式清晰，适合在单机 Linux 沙箱场景中作为基础能力使用。

---

## OverlayFS 方案的局限

虽然 `OverlayFS` 很适合当前目标，但它不是没有代价。

### 1. 隔离能力主要仍依赖 namespace/seccomp/cgroup

`OverlayFS` 解决的是文件系统视图和写时复制问题，不直接提供更强的安全边界。真正的安全隔离仍然依赖：

- mount namespace
- user namespace
- pid namespace
- seccomp
- cgroup

也就是说，它是沙箱实现中的一层，不是完整隔离方案本身。

### 2. lowerdir 目录式管理不够“镜像化”

当前文档中的 lowerdir 是目录，例如：

- `/var/synapse/layers/python-base`
- `/var/synapse/layers/python-scientific`

这种方式实现简单，但在版本管理、完整性校验、发布回滚和内容寻址方面，不如真正的只读镜像格式。

### 3. 对底层文件系统和内核行为有依赖

`OverlayFS` 的行为、限制和性能特征与底层内核版本、宿主文件系统有较强关系。它适合 Linux 专用方案，但不适合追求跨平台一致性的实现。

---

## 有没有更好的方案

有，但“更好”取决于目标是什么。

---

## 方案对比

### 方案 A：继续使用 OverlayFS

#### 优点

- 实现简单
- 启动快
- 支持共享只读层
- 支持每实例独立可写层
- 很适合当前单机 Linux 沙箱模型

#### 缺点

- lowerdir 目录管理相对粗糙
- 安全隔离能力本身有限

#### 结论

对于当前 Synapse 的 MVP 目标，这是合理且务实的方案。

---

### 方案 B：EROFS 或 SquashFS 只读镜像层 + OverlayFS 可写层

这是我认为比当前文档方案更好的演进方向。

#### 思路

将共享运行时从“普通目录 lowerdir”升级为“只读镜像层”，例如：

- `EROFS`
- `SquashFS`

然后在其上继续叠加 `OverlayFS` 的可写层。

结构变成：

1. 只读镜像层：Python runtime / Node runtime / 常用库
2. OverlayFS upperdir：用户代码、临时文件、执行期间改动

#### 优点

- 只读层更真正不可变
- 更适合做版本化管理
- 回滚和发布更清晰
- 通常磁盘利用率更好
- 仍然保留 OverlayFS 的快速写时复制能力

#### 缺点

- 镜像构建链路更复杂
- 运维和发布流程需要额外设计

#### 结论

如果 Synapse 从 MVP 进入稳定迭代阶段，我建议优先演进到这个方案，而不是彻底替换掉 `OverlayFS`。

---

### 方案 C：每次复制完整运行时

#### 优点

- 逻辑直观
- 不依赖 overlay 挂载语义

#### 缺点

- 启动慢
- IO 开销大
- 并发时内存和磁盘压力很高

#### 结论

不适合当前项目目标，应排除。

---

### 方案 D：tmpfs 中展开完整运行时

#### 优点

- 访问速度快

#### 缺点

- 内存占用极高
- 并发时几乎不可控

#### 结论

不适合 200 并发这类目标，应排除。

---

### 方案 E：btrfs / zfs snapshot

#### 优点

- snapshot 能力强
- 回滚和克隆语义好

#### 缺点

- 对宿主文件系统依赖强
- 运维复杂度更高
- 通用部署环境兼容性不如 `OverlayFS`

#### 结论

可以作为特定部署环境下的增强方案，但不适合作为当前默认设计。

---

### 方案 F：MicroVM / 更强虚拟化隔离

典型方案包括：

- Firecracker
- Kata Containers
- gVisor（严格来说不是 MicroVM，但属于更强隔离运行时路线）

#### 优点

- 安全隔离边界更强
- 更适合高安全要求的多租户场景

#### 缺点

- 启动成本更高
- 资源开销更大
- 工程复杂度显著增加

#### 结论

如果目标变成“隔离性优先”，这是更强方案；但如果目标仍是 `product.md` 中强调的极低延迟和轻量执行，则不适合作为当前主方案。

---

## 对 Synapse 的建议

### 短期建议

保留当前 `OverlayFS` 方案。对于 Synapse 当前阶段，它在以下几个方面最平衡：

- 启动时延
- 共享运行时能力
- 实现复杂度
- Linux 单机场景可落地性

### 中期建议

将当前“目录式 lowerdir”升级为：

- `EROFS` 或 `SquashFS` 只读运行时镜像
- 上层继续配合 `OverlayFS`

这会比当前方案更适合：

- 运行时版本管理
- 发布回滚
- 完整性校验
- 降低层内容漂移风险

### 长期建议

如果未来业务场景要求更强的多租户安全隔离，再考虑引入：

- Firecracker
- Kata
- gVisor

但这应当被视为“安全架构升级”，而不是单纯的“文件系统方案替换”。

---

## 最终结论

对 Synapse 而言，`OverlayFS` 不是最先进的方案，但很可能是当前阶段最合理的方案。

更准确地说，最佳路径不是“抛弃 OverlayFS”，而是：

1. 当前阶段使用 `OverlayFS` 完成高并发、低延迟沙箱 MVP
2. 后续将共享运行时层升级为 `EROFS/SquashFS` 等只读镜像
3. 只有在隔离强度成为第一优先级时，再考虑 MicroVM 路线

因此，当前结论是：

- 现在使用 `OverlayFS` 是合理的
- 有更好的演进方案，但不是简单替换
- 推荐方向是“只读镜像层 + OverlayFS”，而不是“放弃 OverlayFS”
